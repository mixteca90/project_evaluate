## 0. 일정 리스크 및 대응 원칙

24시간 이내 구축은 매우 타이트합니다. 아래 원칙으로 진행합니다.

- **집계 엔진(F6/F7)과 P2~P4(채점 제출 흐름), I1 마감 처리를 최우선 확보**합니다. 이 세 가지가 없으면 평가 자체가 성립하지 않습니다.

- 매 단계 완료 시마다 실제 배포본에서 즉시 확인(로컬 개발 서버로 끝내지 않음) — 마지막에 몰아서 배포 이슈를 만나면 복구할 시간이 없습니다.

## 1. 기술 스택 (경량 추천)

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | **Next.js (App Router) + TypeScript** | 프론트+백엔드(API Routes/Server Actions)를 한 프로젝트로 처리, 스캐폴딩이 가장 빠름 |
| DB | **Supabase (Postgres 무료 티어)** | Vercel 서버리스 환경에서 파일 기반 DB(SQLite)는 인스턴스 간 영속성이 보장되지 않음. Supabase는 즉시 발급 가능하고 대시보드에서 데이터를 직접 열람·수정할 수 있어 디버깅이 빠름 |
| DB 접근 | **supabase-js 직접 호출** (ORM 생략) | Prisma 마이그레이션 설정할 시간이 없음. 타입은 손으로 짧게 정의 |
| 인증 | 별도 로그인 시스템 없음 — **이름 선택(명단 시드) + 강사 비밀번호(환경변수 비교)** | OAuth 등 구축 시간이 없고 요구사항상 불필요 |
| 배포 | **Vercel** | git push 시 자동 배포, Next.js와 기본 통합, 무료 |
| 스타일 | **Tailwind CSS** | 클래스만으로 모바일 UI 빠르게 조립 |
| 상태관리 | 별도 라이브러리 없이 fetch + useState/useEffect | 화면 수가 적어 React Query 등 도입이 오히려 시간 손실 |

## 2. 데이터 모델 (Supabase 테이블)

`앱_기능맵.md` 4장의 엔터티를 그대로 테이블로 매핑합니다.

| 테이블 | 주요 컬럼 | 비고 |
|---|---|---|
| `users` | id, name, role(`student`/`instructor`), group_id(학생만) | 시드 스크립트로 16행 고정 입력, 런타임 생성 없음 |
| `groups` | id, name, leader, members(text[]), topic | `조별 명단.txt` 기준 3행 시드 |
| `evaluations` | id, evaluator_id, group_id, item_scores(jsonb, 13개), comment, submitted(bool), anon_label(int), is_locked(bool), submitted_at, updated_at | `item_scores`는 P3 선택 즉시 upsert(F3), `submitted`는 F5 제출 시 true |
| `aggregated_results` | group_id, per_item_final_score(jsonb, 13개), total_score, rank, trimmed_evaluator_ids(int[]), status(`FINAL`) | **마감 전에는 이 테이블에 쓰지 않음** — 잠정치는 API가 evaluations를 매번 즉석 계산. F13 실행 시 1회만 insert |

인증 비밀번호는 `INSTRUCTOR_PASSWORD` 환경변수(Vercel 프로젝트 설정)로 관리하고 코드에 평문으로 넣지 않습니다.

## 3. 화면 → 라우트 매핑

| 페이지 | 라우트 | 비고 |
|---|---|---|
| P0 유의사항 | `/` | 세션에 동의 플래그 저장 후 P1로 |
| P1 본인 확인 | `/login` | 학생 15명 리스트 + "강사로 입장"(비밀번호 모달) |
| P2 평가 대상 조 목록 | `/groups` | role별 필터링(F2) |
| P3 조 정보+채점 | `/groups/[groupId]/score` | F3 자동 임시저장 |
| P4 의견 입력 | `/groups/[groupId]/comment` | F5 제출 검증(13개 항목 완료 여부) |
| P5 전체 결과 | `/results` | F6/F7 호출, 잠정/확정 배지. **강사 전용**(학생 접근 시 본인 조 P6으로 리다이렉트, 2026-07-03 반영) |
| P6 조별 상세 | `/results/[groupId]` | 세부항목 + 익명 의견. **학생은 본인 소속 조만**(타 조 groupId 접근 차단), 강사는 전 조 |
| I1 강사 관리 | `/admin` | 비밀번호 인증 가드, 매트릭스 + 마감 처리 버튼 |

세션은 쿠키에 `{userId, role, groupId}`만 저장(로그인 시스템 없이 단순 세션 쿠키로 충분).

## 4. 핵심 로직 구현 순서 (집계 엔진, F6/F7)

가장 리스크가 큰 부분이므로 **UI와 완전히 독립된 순수 함수**로 먼저 구현하고 단위 테스트로 검증한 뒤 화면에 연결합니다.

```
calcGroupResult(evaluations: Evaluation[], instructorEval: Evaluation | null): {
  totalScore: number | "강사 평가 대기 중",
  perItemScore: number[13] | "강사 평가 대기 중",
  trimmedEvaluatorIds: number[],
  tieBreakApplied: boolean,
}
```

내부 순서 (`조별_프로젝트_평가기준.md` 3-2, `앱_기능맵.md` 3장과 동일):
1. `submitted=true`인 학생 평가만 필터
2. 학생 5인 미만 → 절사 생략(전원 반영) / 5인 이상 → 총점 기준 정렬, 최고·최저 2인 절사(경계 동점 시 evaluator id 오름차순 보조 기준)
3. 절사에서 제외된 평가자를 총점·13개 세부항목 계산 모두에서 동일하게 제외
4. 강사 미제출 → `"강사 평가 대기 중"` 반환하고 종료
5. `(Σ학생점수 + 강사점수×3) ÷ (유효 학생 수 + 3)` — 총점과 13개 항목 각각 동일 공식 적용, **중간값 반올림 금지**
6. 반환 직전(화면 표시/확정 저장 시)에만 소수 둘째 자리에서 반올림해 첫째 자리까지 표시

이 함수에 대해 최소 아래 케이스는 유닛 테스트로 확보합니다: 정상 10명 제출, 경계 동점 2건 이상, 학생 제출 5인 미만, 강사 미제출, 13개 항목 합산=총점 산출식 일치 검증.

## 7. MVP 컷라인 (시간 부족 시 이 순서로 생략)

1. P0 유의사항 페이지 → 정적 텍스트 배너로 단순화(동의 버튼만 유지)
2. I1 매트릭스의 절사 경계 동점 표시 UI → 뱃지 생략, 데이터는 정상 계산(안내 문구만 생략)
3. P3 자동 임시저장의 "이탈 후 복원" → 세션 내 임시저장만 유지, 완전한 새로고침 복원은 생략 가능(단, 데이터 유실 없어야 함)
4. P6 익명 라벨의 "제출 순서 고정" → 단순 조회 시점 순번으로 대체 가능(단, 실명 노출은 절대 불가 — 이 항목은 컷 대상 아님)
5. 진행 배지(N/M 완료) 실시간 갱신 → polling 대신 수동 새로고침으로 대체

**컷 불가(반드시 유지)**: 자기 조 채점 제외, 절사·가중평균 계산 정확성, 강사 3배 가중치, 마감 후 잠금, 익명 처리(P6), 강사 비밀번호 인증.

## 8. 배포 검증 체크리스트

- [ ] 학생 계정으로 로그인 → 본인 조 목록에서 제외 확인
- [ ] 13개 항목 미선택 상태로 제출 시도 → 차단되는지 확인
- [ ] 동일 조에 5명 미만 제출 상태에서 결과 조회 → 절사 생략되고 정상 평균 나오는지 확인
- [ ] 강사 미제출 상태에서 결과 조회 → "강사 평가 대기 중" 노출 확인
- [ ] 강사 계정으로 I1 접근 → 11명(타 조 10명+강사 1명) 미충족 시 마감 버튼이 막히는지 확인
- [ ] 11명 전원 제출 후 마감 처리 → P5/P6이 "확정 결과"로 전환되고 이후 값이 고정되는지 확인
- [ ] P6에서 실명이 아닌 익명 라벨만 노출되는지 확인 / I1에서는 실명 노출되는지 확인

## 9. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| Supabase 초기 설정(테이블·RLS) 지연 | RLS는 비활성화하고 서버 API 라우트에서만 DB 접근(클라이언트에서 직접 Supabase 호출 금지)하여 설정 단계 최소화 |
| 집계 로직 버그로 마감 후 잘못된 확정 저장 | F13 실행 전 I1 화면에서 잠정치를 강사가 육안 확인할 수 있게 하고, 확정 저장 후에도 `aggregated_results` 재계산·재저장이 가능한 관리자 전용 복구 경로를 하나 남겨둠(운영 중 발견되는 버그 대비) |
| 시연/제출 당일 다인원 동시 제출 시 경합 | Supabase의 단일 upsert(evaluator_id+group_id unique 제약)로 동시성 문제 회피 — 별도 락 로직 불필요 |
| 시간 초과 | 6시간 경과 시점(1~4단계 완료 목표)에 진행 상황 점검, 지연되면 즉시 7장 컷라인 적용 여부를 재판단 |
